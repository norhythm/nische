# nische.jp 運用ガイドスライド 設計書

## 概要

クライアント（菊地司氏）へのサービス引き渡し時に使用するレクチャー用スライドを、Marp (Markdown) で作成しPDFとして出力する。

## 対象者

- 菊地司氏（音響エンジニア）
- HTMLを少々扱える
- GitHub / Vercel は未経験
- Pages CMS は操作経験あり（作品の公開設定変更、作品追加を実施済み）

## 形式

- **ツール:** Marp (Markdown Presentation Ecosystem)
- **出力:** PDF (1ファイル)
- **枚数:** 約20枚
- **言語:** 全文日本語。技術用語は平易な表現に置き換える
- **スクリーンショット:** プレースホルダー + 撮影指示コメントを配置。開発者が後からキャプチャを差し込む

## 用語マッピング

スライド内では以下の表現を使用する:

| 技術用語 | スライド内の表現 |
|---------|----------------|
| リポジトリ | ファイル置き場 |
| デプロイ | サイトへの反映 |
| commit | 変更の保存 |
| push | アップロード |
| ビルド | サイトの再構築 |
| frontmatter | 作品情報ヘッダー |
| Markdown | テキストファイル |
| CI/CD | 自動反映の仕組み |
| DNS | ドメインの設定 |
| MXレコード | メールの宛先設定 |
| Aレコード | ウェブサイトの宛先設定 |

## スライド構成

### 表紙 (1枚)

- タイトル: 「nische.jp 運用ガイド」
- サブタイトル: 作成日、対象者

### セクション1: 全体像 (2枚)

**スライド1-1: サイトの仕組み**
- GitHub (ファイル置き場) → Vercel (自動反映) → nische.jp (公開サイト) のフロー図
- 「ファイルを置くだけで自動でサイトに反映される」というメッセージ

**スライド1-2: 各サービスの役割**
- GitHub: 作品データや画像の保管場所
- Vercel: サイトを公開・表示するサービス
- Pages CMS: 作品データを編集する画面
- さくらインターネット: ドメイン (nische.jp) とメールの管理

### セクション2: GitHub — ファイル置き場 (4枚)

**スライド2-1: ログインとトップ画面**
- [スクリーンショット] GitHub ログイン後、nische リポジトリのトップページ
- 撮影指示: `github.com/<菊地氏のアカウント>/nische` のメインページ

**スライド2-2: フォルダ構成**
- `_works/` → 作品のテキストファイルが入っている場所
- `public/works/media/` → 作品画像が入っている場所
- `data/` → プロフィール・機材・連絡先の情報
- それ以外のフォルダ → 基本的に触らない

**スライド2-3: ファイルの中身を見る**
- [スクリーンショット] `_works/` 内の任意の .md ファイルを開いた状態
- 撮影指示: 任意の作品 .md ファイルのプレビュー画面

**スライド2-4: ファイルの編集（直接編集する場合）**
- [スクリーンショット] 鉛筆アイコン (Edit) をクリックした編集画面
- 編集 → "Commit changes" で保存 → 自動でサイトに反映される
- 撮影指示: .md ファイルの編集モードと Commit changes ダイアログ
- 注: 通常はPages CMSから操作するため、これは緊急時用

### セクション3: Vercel — サイトの公開 (3枚)

**スライド3-1: ダッシュボード**
- [スクリーンショット] Vercel のプロジェクトダッシュボード
- 「Ready」= サイトが正常に動いている
- 撮影指示: Vercel ダッシュボードの nische プロジェクトページ

**スライド3-2: 反映の確認**
- [スクリーンショット] Deployments 一覧
- GitHubに変更を保存すると、ここに新しい行が追加される
- 緑のチェック = 成功、赤いバツ = 失敗
- 撮影指示: Deployments タブ、成功したデプロイが並んでいる状態

**スライド3-3: 失敗したとき**
- 赤いバツが出た場合 → 開発者に連絡
- 自分で直そうとしなくてOK
- 連絡先: [開発者の連絡先]

### セクション4: Pages CMS — 作品の管理 (3枚)

**スライド4-1: Pages CMSの画面**
- [スクリーンショット] Pages CMS の作品一覧画面
- 撮影指示: Pages CMS にログインし、_works のファイル一覧が表示された状態
- ※菊地氏は操作経験あり。復習として掲載

**スライド4-2: 作品の追加・編集**
- 新規作成: 「New」ボタン → 各項目を入力 → 「Save」
- 編集: 作品をクリック → 内容を変更 → 「Save」
- 画像: media フォルダに画像をアップロードし、ファイル名を指定

**スライド4-3: 公開 / 非公開の切替**
- `published: true` → サイトに表示される
- `published: false` → サイトに表示されない（下書き状態）
- 保存 → 自動でサイトに反映される

### セクション5: 作品データの構造 (3枚)

**スライド5-1: 作品ファイルの中身**
- frontmatter (作品情報ヘッダー) の各項目:
  ```
  title: 作品名
  artist: アーティスト名
  holder: 権利者・レーベル名
  date: 発売日 (例: 2025-07-15)
  image: 画像ファイル名
  tag: [rec, mix, master] から選択
  published: true または false
  url: ページのURL (英語、ハイフン区切り)
  ```

**スライド5-2: タグの意味**
- `rec` → Recording (レコーディング)
- `mix` → Mixing (ミキシング)
- `master` → Mastering (マスタリング)
- 複数指定可: `tag: [rec, mix]`

**スライド5-3: 画像の置き場所と命名**
- `public/works/media/` フォルダに画像を配置
- ファイル名は作品名と対応させる
- 推奨: JPG、横幅512px以上

### セクション6: DNS・メール — 触らない設定 (2枚)

**スライド6-1: さくらの管理画面で変更禁止の項目**
- MXレコード (`10 www3493.sakura.ne.jp.`) → メールの宛先。**絶対に変更しない**
- Aレコード (`216.150.1.1`) → ウェブサイトの宛先。**変更する場合は必ず開発者に相談**
- [スクリーンショット] さくらインターネットのDNS設定画面
- 撮影指示: さくらの DNS ゾーン編集画面 (MXレコードとAレコードが見える状態)

**スライド6-2: もしメールが届かなくなったら**
- まず確認: さくらの管理画面でMXレコードが `10 www3493.sakura.ne.jp.` になっているか
- 変わっていた場合 → 元に戻す
- わからない場合 → すぐ開発者に連絡

### セクション7: 困ったとき (2枚)

**スライド7-1: よくあるトラブル**
- 「サイトに作品が反映されない」→ Vercelダッシュボードでデプロイ状態を確認
- 「画像が表示されない」→ ファイル名とfrontmatterの `image:` が一致しているか確認
- 「ページが404になる」→ `published: true` になっているか確認、`url:` が正しいか確認

**スライド7-2: 連絡先**
- [開発者の連絡先]
- 「壊れたかも」と思ったらすぐ連絡してください
- 自分で直そうとして悪化するより、連絡してもらった方が早く解決できます

## デザイン

- **カラー:** モノトーン基調 (サイトのトーンに合わせる)。背景: 白、テキスト: #222、アクセント: #888
- **フォント:** Noto Sans JP (本文)、monospace (コード部分)
- **レイアウト:** 1スライド1メッセージ。余白を十分に取る
- **スクリーンショット枠:** グレーの破線ボーダー + 中央に撮影指示テキスト

## ファイル構成

- `docs/slides/operation-guide.md` — Marp ソースファイル
- `docs/slides/operation-guide.pdf` — PDF出力 (Marp CLI で生成)

```bash
npx @marp-team/marp-cli docs/slides/operation-guide.md --pdf --allow-local-files
```
